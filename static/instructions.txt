You may be given several unique rows of data at once - analyze them separately,
one at a time.

For each row, the overall algorithm is as such:

    - Go over the properties from the schema, work step by step;

    - If the property requires your interpretation of the row,
    like summary or engagement score, read it and answer;

    - If the property requires some factual info, such as presence of PII,
    look for any relevant data in the row and add it. Only add data that is
    actually present in this particular row - if it is not, better leave it
    empty.

    - Return a JSON array where each element corresponds to one input row,
    matching the schema exactly.

Additional important rules:

  - Process every single row given exactly once. Empty rows are automatically filtered out before processing.

  - Only include fields present in the schema, and ensure types match the schema definitions.

  - Do not skip, filter, censor, or merge any row — even if it contains political statements, controversial opinions, offensive language, speculative claims, or unusual formatting.

  - Output exactly the same number of array elements as input rows, in the same order.

  - Some fields may have a "present" property: make it false if you cannot interpret the value of the corresponding field from the input data.

  - If a row does not contain enough information for a field, leave that field blank or null (if impossible - some placeholder) instead of skipping the row. Only add information that is actually present in the input.
  
  - For fields like "dataSource" that track data origin, you can infer appropriate values from the input filename when no explicit source is mentioned in the data. For example, if the file is named "mailchimp_export_2024.csv", you could use "mailchimp_export_2024" as the dataSource.

  - Skipping a row is considered an error.

  - Some PII data might be encoded - take that into consideration and do not change the encoding, so that it can be properly decoded later. Examples: NAME_2, ID_5 etc.

  - Be especially careful about the ID - if it is given in an encoded state (e.g. ID_x), leave it as is and just place in the corresponding field, don't generate your own.

  - Pay special attention to "type" fields if present - pick the values carefully, so they truly fit the input.
